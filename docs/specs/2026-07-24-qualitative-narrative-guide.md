## 1. 문제 — 지금 나오는 건 정성조사가 아니다

라이브 가이드(`p_020fcfb16a44`) 실측:

- **Q1 이 예/아니오 질문**이다("최근 3개월 동안 구입한 적이 있나요?"). `GUIDE_SYSTEM` 에
  `예/아니오로 끝나는 닫힌 질문은 피하고` 라고 적혀 있는데 그대로 통과했다 —
  F8 품질 점검(`api/services/evals.py:90`)은 유도신문·버킷 MECE 만 보고 닫힌 질문은 안 잡는다.
- **Q2~Q7 여섯 개가 같은 보기 7개**(브랜드/맛·풍미/영양정보/가격/포장디자인/성분/기타)를
  공유한다. `먼저 확인`·`영향`·`불만`·`만족`·`장애`·`매력` 으로 라벨만 바꾼 같은 격자다.
  Q2↔Q3, Q4↔Q6, Q5↔Q7 은 사실상 동어반복이다.

즉 **음성으로 읽어주는 객관식 설문**이다.

### 왜 이렇게 되는가

1. **버킷이 데이터보다 먼저 온다.** `GUIDE_SYSTEM` 이 문항마다 MECE 버킷 4~7개를 *측정 전에*
   만들라고 요구한다. 정성조사의 코드는 데이터에서 올라오는데(귀납), 보기표를 먼저 깔고 시작한다.
2. **그 버킷이 질문을 지휘한다.** `api/interview/prompts.py:122` 가 진행자에게
   *"직전 답변이 어느 버킷에도 또렷이 안 들어가면 그 자체가 probe 신호"* 를 넣는다.
   파고들기의 목적이 **분류 가능한 답을 받아내는 것**이 된다.

엔진의 정성 장치(래더링 6종·모순 확인·피로 감지)는 멀쩡하다. **가이드가 그걸 못 쓰게 만든다.**

## 2. 결정

### 2.1 질문은 사건 재구성형으로

- 랭킹·빈도 문형(`가장 X한 것은?`, `~한 적이 있나요?`)을 **금지**한다.
- 기본형은 **최근 사건 한 건을 떠올리게 하고 그 장면을 말하게 하는 것**:
  "가장 최근에 ~하셨던 때를 떠올려 볼게요. 어디서, 어떤 상황이었나요?"
- 문항끼리 **서로 다른 국면**을 맡는다(같은 격자 반복 금지): 사건 → 계기·선택의 순간 →
  갈등·포기했던 경험 → 대안과의 비교 → 기대와 어긋난 지점.

### 2.2 버킷은 인터뷰가 끝난 뒤 전사에서 만든다

| | 지금 | 바뀜 |
|---|---|---|
| 생성 시점 | 가이드 생성(측정 전) | **인사이트 생성(측정 후)** |
| 근거 | 모델의 사전 지식 | **실제 응답 전사** |
| 분류 시점 | 인터뷰 중 `reflect_bucket` | 인사이트 생성 시 일괄 |
| 프로빙 영향 | 버킷 적합도가 probe 신호 | **없음** — 프로빙은 이야기만 좇는다 |

- 코드북은 `Insight.codebooks: {question_id: [ResponseBucket]}` 에 저장한다(신규 필드).
- **계약 1 유지**: LLM 은 코드 *생성*과 *분류*만, 개수는 DB 실측(`store.bucket_distribution`).
- 가이드에서 `response_buckets` 편집 UI 를 걷어낸다. 모델 필드는 남긴다(구 데이터 호환).

### 2.3 기존 데이터

- 라이브 가이드의 버킷 47개는 **재활용하지 않는다.** 귀납 코딩의 취지에 반한다.
- 결과 화면은 라벨을 `insight.codebooks` 에서 먼저 찾고, 없으면 기존 가이드 버킷으로
  폴백한다 — **기존 인사이트가 그대로 렌더된다.**
- 기존 `turns.bucket_id` 는 지우지 않는다. 인사이트를 다시 생성하면 새 코드북으로 덮인다.
