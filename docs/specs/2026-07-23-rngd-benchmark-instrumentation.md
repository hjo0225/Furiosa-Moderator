## 0. 한 줄 요약

> 이 벤치마크는 "누가 토큰을 빨리 뽑나"가 아니라 **"이 워크로드에서 지연과 에너지가 어디서 나오는가"**를 구한다.
> 대조군이 없으므로 "몇 배 빠르다"를 말하지 않는다. 대신 **① 동시성을 올려도 안 무너진다 ② 병목은 카드가 아니라 파이프라인이다 ③ 총 에너지는 가동률이 정한다** 세 가지를 실측으로 보인다.

---

## 1. 지표 정의

### M1. 카드당 SLA 충족 동시 세션 수

```
M1 = max{ C : turn_e2e_p95(C) <= 2000ms }
```

- `C` = 동시 진행 세션 수 (카드 1장 기준)
- `turn_e2e` = 참가자 답변 수신 시각 → 모더레이터 발화 **마지막 토큰** 시각 (ms)
- 탐색: 이분 탐색. 하한 1, 상한은 첫 위반 지점의 2배. 해상도 ±4 세션
- 각 `C`마다 워밍업 3분 제외 후 10분 측정
- **함께 기록**: `ttft_p95` (타이핑 인디케이터 체감 지표라 별도 보고), `turn_e2e_p50`

> **주의**: `C`는 "생성 중인 요청 수"가 아니라 "세션 슬롯 수"다. 참가자 사고 시간 때문에 실제 동시 생성은 훨씬 적다. 이 차이가 이 벤치마크의 핵심이므로 절대 혼동하지 말 것.

**SLA 미달 시 보고 방식(필수).** 전 구간에서 `p95 > 2000ms`면 `M1 = 0`이 아니라
**`M1 = 미달(현 아키텍처)`** 로 적고, 대신 **지연 곡선 전체**(C별 p50·p95·실패율)를 보고한다.
"0장으로도 못 한다"와 "카드는 여유가 있는데 파이프라인이 못 따라간다"는 전혀 다른 결론이고,
후자가 실제로 관측된 것이기 때문이다. 이때 §4의 **동시 생성 수**(아래 M1-b)를 반드시 함께 낸다 —
둘을 같이 봐야 원인이 용량인지 파이프라인인지 갈린다.

**M1-b. 평균 동시 생성 수 (슬롯 ≠ 생성).** Little의 법칙으로 산출한다.

```
avg_concurrent_generating = throughput_turns_per_sec * mean_turn_latency_sec
occupancy = avg_concurrent_generating / C
```

참가자 사고 시간(로그노멀) 때문에 슬롯 수보다 훨씬 작다. **이 벤치마크의 고유 논점이므로
M1과 반드시 짝으로 보고한다.** 원자료로 검증 불가능한 "순간 최대 동시 생성"은 보고하지 않는다
(턴별 벽시계 타임스탬프가 없으면 계산할 수 없다 — §4 수집 항목 참조).

### M2. 세션당 Wh — **카드 센서 기준**

```
E_cards    = ∫ P_card(t) dt          over 측정 윈도우   [Wh]
Wh_session = E_cards / completed_sessions

# 분해 (보고 필수)
E_idle    = P_idle_measured * window_hours * N_cards
E_active  = E_cards - E_idle
```

- `P_card` = `furiosa_npu_hw_power` 카드 센서 (W). **CPU·NIC·팬·전원공급 손실 미포함.**
- **보고 시 반드시 "카드 센서 기준 하한값"이라고 명시한다.** 실제 벽면 소비는 이보다 크다.
  이 수치로 전기요금을 계산하거나 타 플랫폼과 비교하지 않는다.
- `P_idle_measured` = 부하 0, 서버 프로세스 기동 상태에서 10분 측정 평균
- **핵심 산출물은 절대값이 아니라 `idle 비중`이다** — 하루 세션 수를 바꿔가며 세션당 Wh가
  어떻게 변하는지의 곡선. 이 워크로드는 duty cycle 이 낮아 총 에너지를 유휴 전력이 지배하며,
  그 사실은 카드 센서만으로도 성립한다(비중은 분자·분모에 같은 편향이 걸려 상쇄되는 방향).

> **벽면 PDU 측정은 범위 밖이다(§9).** 공유 팟이라 물리적으로 접근할 수 없다. 초판은 벽면을
> 필수로 요구했지만 그 요구를 유지하면 M2 를 영원히 보고할 수 없다. 카드 기준으로 내리되
> **한계를 같은 크기 글씨로 적는 것**이 정직한 타협이다.

### M4. 턴 내부 지연 분해

```
turn_e2e = t_emotion + t_generate + t_guardrail   (직렬 구간 기준)
```

- 한 턴을 구성하는 LLM 호출을 단계별로 계측한다. 최소 축: **감정 태깅 · 질문 생성 ·
  가드레일(판정+재작성)**. 각 단계 p50 과 턴 대비 비중을 보고한다.
- **가드레일은 재작성 발생률을 함께 낸다.** 판정만 도는 경우와 재작성까지 가는 경우의
  지연이 다르므로 평균 하나로 뭉개면 병목이 안 보인다.
- 병렬화된 단계가 있으면 벽시계 기준 기여도로 보고한다(직렬 합과 다르다는 점 명시).

> M1 이 "얼마나 버티나"라면 M4 는 **"어디서 시간을 쓰나"**다. 대조군이 없는 상황에서
> 가장 실행 가능한 결론을 주는 지표라 M1·M2 와 동급으로 승격했다.

## 2. 헤드라인: idle 지배 분석

손익분기 세션 수 `S*`는 **범위 밖이다(§9)** — GPU/클라우드 대조군 비용과 벽면 전력이 둘 다
없으면 교차점을 계산할 수 없다. 추정 상수로 곡선을 그리면 그 순간 실측 문서가 아니게 된다.

대신 **실측만으로 성립하는 같은 논증**을 낸다: 세션당 에너지는 하드웨어가 아니라 **가동률**이 정한다.

```python
# 하루 세션 수 S 에 대한 세션당 에너지 (카드 센서 기준)
E_idle_24h   = P_idle_measured * 24 * N_cards          # Wh
E_active(S)  = S * Wh_active_per_session               # Wh
Wh_per_session(S) = (E_idle_24h + E_active(S)) / S
idle_share(S)     = E_idle_24h / (E_idle_24h + E_active(S))
```

**출력 형태**: 하루 세션 수 × {세션당 Wh, idle 비중} 표 + 곡선. 하한/상한 세션 수 사이에서
세션당 에너지가 몇 배 차이 나는지를 헤드라인으로 삼는다.

> 이렇게 하면 "월 몇 세션부터 유리한가"라는 원래 질문을 **비용 비교 없이** 던질 수 있다 —
> "유리해지려면 가동률이 얼마나 필요한가"로 바꿔 묻는 것이고, 답은 실측에서 나온다.

## 3. 부하 재생기 요구사항

### 세션 구조 (부록 A 기준)

```
세션 시작
  → 인사/동의
  → Q1..Q7 (Q5는 척도형)
      각 질문마다 프로빙 0~3회 (max_probes는 질문별 상이)
  → 클로징
세션 종료
  → 버킷 분류 (세션 전체 답변)
  → 세션 요약
```

- 모더레이터 발화 턴 수: 세션당 18~25
- 출력 길이: 턴당 60~100 토큰

### 컨텍스트 구성 — **캐시 히트를 좌우하므로 순서 고정 필수**

```
[고정 프리픽스 — 캐시 대상]
  1. 시스템 프롬프트 (바이트 단위 고정)
  2. 모더레이터 페르소나
  3. 인터뷰 가이드 JSON
  4. 동결된 지식 팩
[가변 — 캐시 미스 구간]
  5. 누적 트랜스크립트
  6. 참가자 ID / 타임스탬프 등 세션 고유값
```

> **구현 규칙**: 세션마다 달라지는 값은 반드시 5번 이후에 배치한다. Furiosa-LLM의 prefix caching은 토큰 단위 정확 일치라 구두점·공백 하나만 달라져도 프리픽스 전체가 미스 처리된다. 프롬프트 템플릿은 문자열 포매팅이 아니라 **사전 렌더링된 상수**로 관리할 것.

### 도착·지연 분포

| 파라미터 | 분포 | 비고 |
|---|---|---|
| 세션 도착 | 포아송 | λ는 목표 동시 세션 수에서 역산 |
| 참가자 사고+타이핑 지연 | 로그노멀 (median 25s) | 10~12분 / 20턴에서 역산 |
| 답변 길이 | 실제 트랜스크립트 코퍼스에서 샘플링 | 합성 금지 |

### 코퍼스 고정 (필수)

부록 A 가이드 + 실제 테스트 인터뷰 트랜스크립트를 **한 번 생성해 파일로 동결**하고, 하드웨어를 바꿔가며 동일 코퍼스를 리플레이한다. 매 실행마다 새로 생성하면 하드웨어 차이인지 입력 차이인지 구분 불가.

## 4. 데이터 수집

| 소스 | 수집 항목 | 주기 |
|---|---|---|
| PDU (SNMP/HTTP) | 벽면 전력 W | 1s |
| `furiosa-metrics-exporter` → Prometheus | `furiosa_npu_hw_power{label="rms"}`, `furiosa_npu_core_utilization`, `furiosa_npu_core_frequency`, `furiosa_npu_throttling_events_count` | 1s (`--interval 1`) |
| furiosa-llm `/metrics` | running/waiting requests, KV cache usage | 1s |
| furiosa-llm 서버 로그 | prefix cache hit ratio | 파싱 |
| 부하 재생기 | 턴 단위 타임스탬프 | 이벤트 |

**시각 동기**: 모든 소스에 NTP 동기 필수. 전력 시계열과 턴 이벤트를 조인해야 하므로 오차 100ms 이내.

  "t_request": "iso8601",
  "t_first_token": "iso8601",
  "t_last_token": "iso8601",

## 5. 실험 매트릭스

| 축 | 수준 | 우선순위 |
|---|---|---|
| 동시 세션 | 이분 탐색 (M1 산출용) | **필수** — 실측 완료 |
| 역할 | 모더레이터 발화 / 프로빙 판정 / 버킷 분류 | **필수** — M4 분해의 축 |
| governor | Performance / PowerSave | 선택 — 고정 부하에서 전력 비교용 |
| prefix caching | on / off (`--no-enable-prefix-caching`) | 선택 — 히트율 로그 파싱이 선행돼야 함(§9) |
| ~~하드웨어~~ | ~~RNGD / GPU 대조군~~ | **범위 밖(§9)** — 대조군 하드웨어 없음 |

**축소 지침**: 전조합을 돌리지 말 것. M1은 `governor=Performance, cache=on`에서만 구하고, governor·cache 축은 **고정 부하(M1의 70%)에서 전력 비교용으로만** 사용한다.

### 모델 배치 (SDK 검증 등급 기준)

| 역할 | 1순위 | 비고 |
|---|---|---|
| 모더레이터 발화 (한국어) | `furiosa-ai/EXAONE-4.0-32B-FP8` | Performance ✅ |
| 프로빙 판정 / 버킷 분류 | `furiosa-ai/Llama-3.1-8B-Instruct` | Performance ✅ |
| 대안 (버킷 분류) | `furiosa-ai/Qwen3-8B-FP8` | Performance 🟡 — 결과에 등급 명시 |

가이드 생성·종합 요약은 측정 대상에서 **제외** (저빈도, 이관 대상 아님).

> **실행 배치는 결과에 반드시 적는다.** 2026-07-23 측정은 위 배치가 아니라 **전 역할 Qwen3-8B**로
> 돌았다(가용 엔드포인트 제약). 모더레이터 발화까지 8B로 돌린 결과의 품질 저하는 그 자체로
> "32B가 왜 필요한가"의 정성 증거지만, **위 배치의 성능을 대변하지 않는다.** 배치가 다르면
> 지연·전력 수치를 같은 표에 섞지 말 것.

## 6. 측정 프로토콜

1. 워밍업 3분 → 폐기
2. 본 측정 10분 × 3회 반복
3. 보고: 중앙값 + p95 + 표준편차
4. `furiosa_npu_throttling_events_count`가 증가한 구간이 있으면 해당 회차 무효 처리 후 재측정

## 8. 구현 시 하지 말 것

- **카드 센서 수치를 벽면 전력인 것처럼 쓰기.** 카드 기준으로 보고하는 것은 허용하되(§1 M2),
  한계 문구를 같은 크기로 붙이고 전기요금 계산·타 플랫폼 비교에 쓰지 않는다.
- 24h 윈도우를 가속 리플레이 (유휴 시간이 지표의 본질)
- 매 실행마다 코퍼스 재생성
- 프롬프트를 런타임 문자열 포매팅으로 조립 (캐시 파괴)
- **원자료로 검증 불가능한 값 보고** — 예: 턴별 타임스탬프 없이 "순간 최대 동시 생성"
- **다른 모델 배치의 수치를 같은 표에 섞기** (§5)
- 확인 안 된 값을 추정치로 채워 넣기 — 미측정 항목은 `null`로 남기고 사유 기록

## 9. 범위 밖 — 못 잰 것과 그 사유

**이 절의 항목은 "아직 안 했다"가 아니라 "이 환경에서는 못 한다"이다.** 결과물에 `null` 로 남기고
사유를 함께 싣는다. 나중에 환경이 바뀌면 여기서 위로 올린다.

| 항목 | 사유 | 되살리려면 |
|---|---|---|
| **벽면 PDU 전력** | 공유 팟 — PDU 물리 접근 불가 | 전용 랙/PDU 확보 |
| **GPU 대조군** | 대조군 하드웨어 미확보 | 동급 GPU 장비 |
| **Cohen's κ (구 M3)** | 골드셋 500건 미구축 + 대조군 없음(Δκ 계산 불가) | 인간 이중 라벨링 500건 |
| **손익분기 S\*** | 대조군 비용·벽면 전력에 의존 | 위 셋 중 둘 이상 |
| **프리픽스 캐시 히트율** | 로그 파싱 미구현 | 파서 구현(작은 작업) |
| **재현성 메타데이터** | firmware/driver/corpus_hash 미기록, 벤치 스크립트가 리포에 없고 팟에만 존재 | 스크립트 리포 이관 |

### κ 게이트를 M1·M2 에서 분리한다

초판 §8은 *"κ 미달 상태에서 M1·M2 보고"* 를 금지했다. 의도는 "분류 품질이 무너졌으면 성능
숫자는 의미 없다"였지만, **κ 를 잴 수 없는 환경에서는 이 조항이 모든 보고를 막는다.**

→ **분리한다.** 지연·에너지는 분류 품질과 독립적으로 성립하는 측정이다. 대신 결과물에
**"분류 품질(κ)은 미검증"** 을 같은 지면에 명시하고, 품질 주장은 정성 관찰(§7 부록)로만 한다.
κ 를 잴 수 있게 되면 게이트를 되살린다.

**진단이 맞았다.** 가드레일이 콜백을 유도로 오판하던 것을 고치자 재작성률이 1/5 로 떨어졌다.

**단, 두 런은 모델·카드 수·코드가 모두 달라 직접 비교가 아니다.** 재작성률은 프롬프트 규칙에
직접 걸리는 값이라 해석이 성립하지만, 지연 수치를 "몇 초 줄었다"로 말하면 안 된다.

또 하나 — **동시성이 올라가면 지연이 무너진다.** 슬롯 1→32 에서 p50 3.07s → 10.92s(3.6배).
구런(8B)의 4.96→5.93s 평탄함과 반대다. 모델이 커져 카드가 실제로 포화된 것으로 보이며,
`슬롯 ≠ 생성` 논증의 전제(카드가 논다)가 이 구성에서는 성립하지 않을 수 있다.
**세션 모드(사고시간 포함) 재측정 없이 이 표를 유휴 지배 논증의 근거로 쓰지 말 것.**

### 재측정이 필요한 시점

파이프라인을 고치면 M4 와 M1 이 같이 바뀐다. **다음 변경 이후의 수치는 이전 수치와 비교 불가다:**

- 감정 태깅 병렬화 (PR #30)
- 가드레일 판정 규칙 수정 — 콜백 오탐 제거 (PR #33)

두 변경 모두 M4 의 단계 비중과 턴 전체 지연을 직접 건드린다. **수정 전 측정치를 현재 성능으로
제시하지 않는다** — 재측정 전까지는 "수정 전 기준"이라고 적는다.
