## 1. 문제 — 라이브 실측

응답 4건짜리 라이브 프로젝트(`p_020fcfb16a44`)를 열어 확인한 것.

1. **버킷 분포가 대부분 비어 있다.** 한 문항의 버킷 7개 중 값이 있는 건 1개, 나머지는
   `0 · 0%` 회색 바가 자리만 차지한다. 화면 세로 절반이 정보 없는 칸이다.
2. **차트가 N 에 비해 과대하다.** `주제별 언급 수`·`감정 분포` 막대차트 2개가 첫 화면을 다
   먹는데 응답은 4건이다. ui-ux-pro-max 차트 DB 도 분포형 차트는 그룹당 20점 미만이면
   패턴이 의미 없다고 명시한다.
3. **비율이 거짓말을 한다.** `2 · 100%` 는 "응답 2건 중 2명"인데 화면만 보면 전원으로 읽힌다.
4. **인용이 중복이다.** "가격이 일반 콜라보다 200원 이상 비싸면 망설인다"가 표현만 다른 채
   4번 반복된다. 응답 4건이라 사실상 전원 인용이다.
5. **한 페이지 세로 적층.** 전체 인사이트 → 차트 2개 → 문항별 요약·분류 → 주제별 인용 →
   응답 목록이 전부 한 스크롤에 있다.
6. **탭이 URL 에 없다.** `project-view.tsx:41` 이 `useState<Tab>` 이라 공유도 뒤로가기도
   안 된다. ui-ux-pro-max UX 규칙 Deep Linking 위반.

## 2. 결과 화면 = 하위 탭 3개

```
[인터뷰 가이드] [결과]
                └─ (요약) (주제별) (응답 4건)
```

`tab` 이 없을 때의 기본값으로만 남긴다.

## 3. 요약 탭 = 내러티브 리포트

문장이 주인공이고 수치는 근거로 인라인에 붙는다. 그대로 의뢰자에게 보낼 수 있는 문서다.

```
가격 민감도가 구매를 가릅니다                         강함

4명 중 3명이 일반 콜라 대비 200원을 임계점으로 집었습니다.

  "200원 이상 비싸면 망설인다"  — 응답자 2

근거 · 가격 3 / 맛 1 / 미언급 0                      (n=4)
```

- 발견 3~5개. 각각 **헤드라인 + 근거 한 문장 + 대표 인용 1개**.
- 상단에 한 문단 총평(기존 `Insight.overall` 을 그대로 쓴다).
- 하단에 모수·수집 시점.
- **차트 없음** (§4 규칙).

## 4. 정직성 규칙 (이 스펙의 핵심)

가독성 문제로 보이지만 실제로는 정직성 문제다. AGENTS.md §0.1 계약 1(집계 숫자는 LLM 이
세지 않는다)의 연장선에 둔다.

| 규칙 | 내용 |
|---|---|
| **작은 N 은 비율을 쓰지 않는다** | `n < 10` 이면 `%` 를 표기하지 않는다. `2 · 100%` → `2명 중 2명`. 분모를 항상 같이 적는다. |
| **작은 N 은 차트를 그리지 않는다** | `n < 10` 이면 막대차트 대신 인라인 카운트. `n >= 10` 이면 기존 차트를 낸다. |
| **0건 버킷은 접는다** | 값 0 인 버킷은 `미언급 5개` 로 접고 펼치기를 준다. 지우지는 않는다 — "안 나온 것"도 정보다. |
| **유사 인용은 하나만** | 같은 취지 인용은 대표 1개. 4명이 같은 말을 하면 `4명이 같은 취지로 답했습니다` + 대표 인용 1개. |
| **수치는 전부 DB 실측** | LLM 은 문장·인용 선택만. `mention_count`·`sentiment`·`bucket_distribution` 은 지금처럼 서버가 덮어쓴다(계약 1). |

**`n` 은 문맥마다 다르다. 화면에서 섞으면 안 된다.**

| 어디 | `n` | 왜 |
|---|---|---|
| 요약 탭의 발견(theme) | **완료 응답 수** (`Insight.session_count`) | theme 은 인터뷰 전체를 가로지른다 |
| 주제별 탭의 질문·버킷 | **그 질문을 실제로 받은 응답자 수** | 턴 배분 때문에 질문마다 다르다 |

지금 화면이 "응답 4건"과 "응답 2건"을 같은 라벨로 쓰면서 왜 다른지 안 알려주는 게 정확히
이 혼동이다.

## 5. 주제별 탭

주제 > 질문 > 버킷·인용. 스펙 A 의 주제 계층을 그대로 쓴다.

- 질문마다 **`4명 중 2명이 이 질문을 받았습니다`** 를 붙인다. 지금 화면의 "응답 2건" 이 왜
  4가 아닌지 알 수 없던 문제를 여기서 해소한다.
- 버킷 분포는 §4 규칙 적용 (0건 접기, 작은 N 비율 금지).
- 질문별 `QuestionSummary.headline` / `summary` 는 그대로 쓴다.

## 6. 응답 탭

지금의 세션별 전사 목록을 유지하고 밀도만 개선한다. CSV·JSON 내보내기 버튼은 이 탭으로 옮긴다
(지금은 전체 인사이트 헤더에 붙어 있어 무엇을 내보내는지 모호하다).

## 7. 데이터 모델 영향

새 필드를 만들지 않고 기존 `Insight` 를 재해석해서 쓴다.

| 기존 필드 | 새 화면에서의 역할 |
|---|---|
| `overall` | 요약 탭 상단 총평 |
| `themes[]` (`theme`·`summary`·`quotes`·`keywords`·`mention_count`) | 요약 탭의 **발견** — `mention_count` 는 이미 DB 실측이라 "4명 중 3명"의 분자로 그대로 쓴다 |
| `question_summaries[]` | 주제별 탭의 질문 헤드라인 |
| `bucket_distribution` | 주제별 탭의 버킷 |
| `sentiment` | `n >= 10` 일 때만 차트 |

**따라서 이 스펙은 API 스키마를 바꾸지 않는다.** 인용 중복 제거는 인사이트 생성 프롬프트
(`api/prompts/insight.py`)에서, 나머지는 전부 표현 계층에서 처리한다.

## 9. 완료 기준 (DoD)

1. `?tab=results&view=topics` 로 직접 들어가면 그 탭이 열린다. 뒤로가기가 탭 사이를 오간다.
2. 응답 4건 프로젝트의 요약 탭에 **차트가 없고**, `%` 표기가 없고, `4명 중 3명` 형태로 적힌다.
3. 값 0 인 버킷이 기본으로 접혀 있고 펼치면 보인다.
4. 같은 취지 인용이 하나로 합쳐지고 `4명이 같은 취지로 답했습니다` 가 붙는다.
5. 질문마다 "몇 명이 이 질문을 받았는지"가 보인다.
6. 라이브 스크린샷으로 전/후 비교 (AGENTS.md §3 Verify).
7. CI 4종 통과.

## 10. 열어둔 것

  응답이 쌓인 뒤로 미룬다 — **검증 안 된 경로임을 알고 넘긴다.**
