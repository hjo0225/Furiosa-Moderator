인터뷰 엔진을 LangGraph로 바꾸는 이유와 방법

10분이면 전체 그림이 잡히도록 만든 팀 공유 문서입니다. 세부 결정 기록은 레포의 LANGGRAPH_PLAN.md에 있습니다.

STEP 1지금의 한계

현재 인터뷰 엔진이 못 하는 것 4가지

지금도 잘 돌아갑니다. 하지만 구조적으로 막혀 있는 게 4개 있어요 — 전부 "더 좋은 인터뷰어"가 되는 길목입니다.

① 생각이 한 덩어리 — "만능 1콜"

파고들지·넘어갈지·끝낼지를 LLM 호출 한 번이 전부 결정합니다. 행동을 조종하려면 프롬프트 문장 순서를 바꾸는 수밖에 없는데, 실제로 문항 목록을 앞에 뒀다가 7턴 내내 대본만 읽은 사고가 있었습니다(코드 주석에 기록됨).

🤯 LLM 콜 하나
답변 분석전략 결정질문 생성종료 판단문항 추적
전부 프롬프트 한 장에 우겨넣음 — 고장나면 원인 추적 불가

② 매 턴 기억을 처음부터 재조립

"기다린다"는 개념이 코드에 없습니다. 요청이 올 때마다 DB에서 대화 전체를 다시 읽어 상황을 재구성하고, 답하고, 전부 잊습니다. 매 턴 새 직원이 서류철 읽고 퇴근하는 구조.

턴 요청→DB 전부 읽기→계산→저장→기억 소멸
턴 요청→DB 전부 읽기→계산→저장→기억 소멸
턴 요청→DB 전부 읽기→…
같은 일을 매 턴 반복 — 인터뷰의 "연속성"이 구조에 없음

③ 행동이 2개뿐

지금 모더레이터가 고를 수 있는 건 "꼬리질문" 아니면 "다음 문항" 뿐입니다. 좋은 인터뷰어가 하는 행동들 — 모순 확인, 애매한 답 되묻기, 얇게 넘어간 문항 재방문 — 을 추가할 자리가 없습니다.

"배달비가 좀 부담돼요…"
꼬리질문다음 문항
모순 확인되묻기재방문주제 복귀
회색 취소선 = 지금 구조에선 불가능한 행동들

④ 의뢰자의 도메인을 모른다

응답자가 "배민클럽 때문에 갈아탔어요"라고 해도 모더레이터는 그게 뭔지 모릅니다. 최악의 반응 — "배민클럽이 뭔가요?"(응답자가 설명하느라 흐름 끊김) 또는 아는 척 헛다리 질문. 도메인 이해 = probing 품질 그 자체인데 참고할 자료가 없습니다.

✗ "배민클럽이요? 그게 뭔가요?" ← 응답자가 설명해야 함 ✗ 아는 척 → 헛다리 질문
✓ 의뢰자 자료를 검색해 아는 인터뷰어 (RAG로 해결)
이 한계만 RAG가 필요 — 나머지 3개는 RAG 없이 해결됩니다
STEP 2LangGraph는 언제 쓰나

LangGraph의 본체 = "실행의 세이브/로드"

병렬처리·분기·재시도는 파이썬 기본기라 프레임워크가 필요 없습니다. LangGraph의 진짜 능력은 실행을 게임처럼 저장했다가 이어하기예요 — 일반 파이썬 함수로는 만들 수 없는 능력입니다.

🕹️ 일반 파이썬 함수
실행 시작 → 끝까지 가거나, 죽거나.
중간에 끄면? 처음부터 다시.
(호출 스택은 저장이 안 됨)

그래서 지금 구조는 매 턴 "짧게 켰다 끄기"를 반복하는 것.

💾 LangGraph
실행 중 어디서든 ⏸ 일시정지
→ 자동 세이브 (체크포인트)
→ 며칠 뒤, 다른 서버에서도 ▶ 이어하기
→ 과거 세이브로 🕰️ 되감기도 가능

"사람을 기다렸다 이어가는 프로그램"을 만들 수 있게 됨.

사람의 입력을 기다렸다 이어가야 한다승인 대기, 답변 대기…
인터뷰가 정확히 이것 ✓
실행이 요청 한 번보다 오래 산다몇 분~몇십 분짜리 흐름
인터뷰는 6~12턴 ✓
같은 루프를 돌면서 상태가 쌓인다질문→답변→질문→…
인터뷰의 본질 ✓
중간 상태를 저장·복원·되감기 하고 싶다크래시 복구, 디버깅
"7턴에 왜 안 캐물었지?" ✓
LLM이 도구를 골라 쓰며 판단한다검색할까? 그냥 답할까?
도구를 만들면 ✓ (STEP 3)
✅ 인터뷰는 "사람을 기다리는 장수명 루프" — LangGraph가 가장 잘하는 문제 모양과 정확히 일치합니다. 억지 적용이 아니에요.
STEP 3그래서 이걸 바꾼다

필수 개조는 딱 3개, 나머지는 선택 확장

LangGraph는 "노드들이 · 공유 상태를 주고받으며 · 오래 사는 실행 안에서" 도는 물건입니다. 지금 코드엔 이 세 가지가 다 없어서, 그 셋을 만드는 게 필수 개조 3개입니다.

필수 ①

매 턴 새 직원 → 인터뷰 전담 진행자 (스레드화)

왜: "기다림"이 구조에 없으면 LangGraph의 간판 기능(일시정지·이어하기)을 쓸 자리가 없습니다. 인터뷰 하나를 살아있는 실행 하나로 만듭니다.

지금턴 요청마다 실행이 새로 태어나고 죽음.
상태는 매번 DB에서 재조립.
→
개조 후세션 시작 = 실행 시작.
질문하고 ⏸ 잠들기 → 답변 오면 ▶ 깨어나기 → 반복 → 종료.
세이브는 자동(체크포인트).

일시정지·이어하기·크래시 복구·과거 되감기가 전부 이 개조 하나로 열립니다.

필수 ②

만능 1콜 → 역할 분담 (노드 분해)

왜: 노드가 1개면 그래프가 아닙니다. 그리고 "생각 덩어리"를 쪼개야 각 판단을 따로 고치고, 따로 테스트하고, 어디서 고장났는지 볼 수 있습니다. LLM 호출 수는 지금과 같아요 — 콜을 늘리는 게 아니라 역할을 나누는 겁니다.

지금🤯 한 콜이 분석+전략+생성+판단 전부
→
개조 후
👂 듣기 — 답변 분석🧭 전략 — 행동 선택🎤 질문 — 생성🛡️ 검수 — 기존 가드레일

"왜 이 질문을 했지?"가 전략 노드의 기록으로 남음. 행동 추가 = 프롬프트 수술이 아니라 분기 하나 추가.

필수 ③

기억을 그래프 안으로 (State)

왜: 자동 세이브(체크포인트)는 "그래프가 들고 있는 상태"를 저장합니다. 상태가 전부 바깥 DB에만 있으면 저장할 게 없어서 세이브 기능이 장식이 됩니다. 대화·커버리지·페이스를 그래프의 State로 옮깁니다. (대시보드용 DB 테이블은 그대로 유지 — 이원화가 의도된 설계)

지금상태가 전부 Postgres 테이블에만.
실행은 아무것도 기억 안 함.
→
개조 후그래프가 State를 소유:
대화 이력 · 커버리지 · 페이스 · 이번 턴 작업 메모리
→ 매 스텝 자동 세이브

잠들었다 깨어나는 "진짜 재개"가 가능해짐. 되감기 디버깅의 재료도 이것.

선택 확장 5개 — 원할 때 하나씩, 전부 미루거나 잘라도 됨

확장

행동 2개 → 7개

모순 확인·되묻기·재방문·주제 복귀 추가. 그래프에선 분기 하나 추가일 뿐.

→ 진짜 인터뷰어다운 대화
확장 · RAG ⭐

브리핑 라이트

의뢰자가 올린 자료를 청크→임베딩→검색(brief). 교과서 RAG 그대로, 문서 입구만 "업로드 전용" (웹 검색 없음).

→ 도메인을 아는 인터뷰어
확장

도구 4종

브리핑 검색(brief)·질문 기법 사전·커버리지 리포트·페이스. 모델이 필요할 때만 호출.

→ 프롬프트에 다 안 넣고 필요한 것만
확장

슬로우패스

응답자가 말하는 20~30초(공짜 시간)에 무거운 분석을 병렬로 처리.

→ 지연 증가 없이 인지 상승
확장

질문 스트리밍

이미 있는 스트리밍 코드(첫 토큰 0.26초 실측)를 연결만 하면 됨.

→ 체감 반응속도 상승
✂️ 이번에 뺀 것: 스터디 메모리(세션 간 장기기억 — 12질문 규모엔 과함) · recall(과거 발화 검색 — 인터뷰 ~24턴이 컨텍스트 창에 통째로 들어가 검색 불필요) · 웹 자동 리서치(브리핑을 업로드 전용으로 축소). RAG는 브리핑 하나로 집중됩니다.
✅ 안심 포인트 — 안 건드리는 것: LLM 클라이언트의 Qwen3 튜닝 · 가드레일 · PII 마스킹 · 음성(STT/TTS) · DB 테이블 · 웹 화면 대부분. 실제로 갈리는 코어는 moderator.py(193줄) + 턴 핸들러입니다.
STEP 4그럼 이렇게 좋아진다

완성된 모습: 한 턴이 도는 길

필수 3개 + 확장을 조립하면 이 그림이 됩니다. 이 다이어그램이 곧 코드 구조예요 — 그래프 도구로 그대로 시각화됩니다.

▼

💬

더 인터뷰어다운 대화

"아까 말씀하신…" 콜백, 모순 확인, 얇은 문항 재방문 — 지금은 불가능한 행동들이 열림.

🕰️

타임머신 디버깅

"7턴에서 왜 안 캐물었지?" → 그 시점 세이브를 열어 전략 노드의 판단 이유를 직접 확인.

⚡

지연은 그대로거나 더 빠르게

무거운 생각은 사람이 말하는 시간에. 목표 첫 반응 1.5~2.5초 (현재 2.4~3.8초).

🔄

끊기지 않는 인터뷰

서버 교체·크래시가 나도 마지막 세이브에서 이어짐 — 응답자는 눈치도 못 챔.

🗺️

눈에 보이는 구조

위 다이어그램이 코드 그 자체. 새 팀원도 그림 한 장으로 엔진을 이해.

LangGraph 핵심 기능 10개 중 실제로 쓰이는 개수. 세션 간 장기기억(Store) 하나만 의식적으로 제외 — 12질문 규모엔 과함. 검수 기준: "장식으로 쓰는 기능이 하나도 없다."

진행 순서

🔵 필수 코어 — 여기까지만 해도 성립

  • 0단계 기반 공사 + Qwen3 도구 선택 30분 실측 (최대 리스크 먼저 제거)
  • 1단계 그래프 골격 — 스레드화 + 노드 분해 (필수 ①②)
  • 2단계 상태 이동 + 커버리지 (필수 ③)

🟡 선택 확장 — 원할 때 하나씩

  • 3단계 행동 7종 확장 (모순 확인·재방문·되묻기)
  • 4단계 슬로우패스 + 질문 스트리밍
  • 5단계 브리핑 라이트 — RAG · 업로드 전용 (+ 도구 4종)
알려진 리스크: Qwen3의 자율 도구 선택 품질이 미검증(0단계 실측이 게이트) · 브리핑 검색에 리랭커 API 형태 확인 필요(안 되면 임베딩 top-k만) · 개인정보는 그래프에 들어가기 전에 마스킹(세이브 파일에 원문이 남으면 안 되므로).  ✂️ recall · 스터디 메모리 · 웹 자동 리서치는 이번 범위에서 제외.
