# 발표 슬라이드 구성안 — AI 중재 인터뷰 플랫폼

| 항목 | 내용 |
|---|---|
| 발표 시간 | 10분 |
| 슬라이드 수 | 13장 |
| 제출물 | pptx (클립 임베딩) + 통합 동영상 파일 |
| 근거 문서 | AI-Moderated-Interview-PRD v1.0 |

## 구성 원칙

**개념 슬라이드를 따로 두지 않는다.** 기능을 보여주면서 그 기능이 왜 그렇게 설계됐는지를 같이 말한다. 데모 영상은 한 덩어리로 몰지 않고 **기능 슬라이드마다 짧은 클립으로 쪼개서** 붙인다.

단, 기능 소개 직전에 **아키텍처 2장**을 둔다. 청중이 "지금 보는 화면이 시스템의 어디인가"를 알고 봐야 네 개의 클립이 따로 놀지 않는다.

## 시간 배분

| 구간 | 슬라이드 | 클립 | 시간 |
|---|---|---|---|
| 문제 · 해법 | 1–3 | — | 1분 50초 |
| **아키텍처** | **4–5** | — | **1분 30초** |

## S1. 타이틀

- 제품명 + 한 줄 정의: "설문의 스케일과 인터뷰의 깊이를 동시에"
- 팀명 / 발표자 / 날짜
- 벤치마크 대상: Outset.ai (YC S23)

**비주얼**: 다크 배경. 채팅 말풍선 모티프 하나만.

## S2. 문제 — 리서처는 두 가지 나쁜 선택지에 갇혀 있다

| | 설문조사 | 심층 인터뷰 |
|---|---|---|
| 표본 규모 | 수백~수천 | 5~15 |
| 소요 기간 | 1~3일 | 3~6주 |
| 인사이트 깊이 | 표면적 (What) | 깊음 (Why) |
| 후속 질문 | 불가능 | 가능 |
| 분석 비용 | 자동 | 수동 태깅 20~40시간 |

**핵심 문장**: "설문은 '왜'를 못 잡고, 인터뷰는 '몇 명이나'를 못 잡는다."

**비주얼**: 좌우 2단 대비. 각 단의 약점 셀을 강조색으로.

---

## S3. 해법 — 네 가지 병목을 각각 어떻게 푸는가

**출발점**: LLM은 강력한 "질문자"다. 토론 가이드만 주면 수백 명과 동시에, 24시간, 일관된 톤으로 심층 대화를 진행할 수 있다.

| 병목 | 해결 메커니즘 | 결과 |
|---|---|---|
| 표본이 5~15명에 묶임 | 링크 배포로 수백 명 동시 진행 | 목표 N 120명 |
| 리크루팅·일정조율에 3~6주 | 24시간 가동, 조율 대상이 없음 | 100건을 24시간 내 수집 |
| 설문은 후속 질문이 불가능 | 프로빙 엔진 | 설문 대비 3배 정보량 |
| 수동 태깅 20~40시간 | 사전 정의 버킷으로 실시간 분류 | 종료 후 5분 내 리포트 |

**핵심 문장**: "리서처를 없애는 게 아니라, 리서처를 로지스틱스에서 해방시킨다."

**하단 캡션 (작게)**: 리서치 설계 · 가설 수립 · 스토리텔링 · 윤리적 판단은 여전히 리서처의 일이다.

**비주얼**: 좌(병목) → 우(해결) 4행 매핑, 행마다 화살표.

## S4. 시스템 아키텍처 — 구성과 데이터 흐름

> 정적 구조. "무엇으로 이루어져 있고 데이터가 어디로 흐르는가"
> **내용은 프로젝트 코드 확정 후 작성**

### 담을 섹션

| 섹션 | 다룰 것 |
|---|---|
| 클라이언트 | 참가자 웹 / 리서처 웹 / 공유 리포트 — 반응형 단일 앱 |
| 애플리케이션 서버 | 스터디·세션 관리, 인증, 링크 발급 |
| LLM 계층 | 역할별 모델 라우팅 (가이드 생성 / 발화 / 판정 / 분류) |
| 검색·저장 | 지식 DB(벡터), 세션·트랜스크립트 저장소 |
| 도메인 모델 | Study → Guide/Pack → Session → Turn → Analysis 관계도 |
| 외부 연계 | 패널사 리다이렉트, Export (CSV/PPTX) |

**비주얼**: 계층형 블록 다이어그램. 화살표는 데이터 흐름 방향만.

## S5. 처리 파이프라인 — 한 턴에서 무슨 일이 일어나는가

> 동적 동작. "요청 하나가 들어오면 어떤 순서로 처리되는가"
> **내용은 프로젝트 코드 확정 후 작성**

### 담을 섹션

| 섹션 | 다룰 것 |
|---|---|
| 가이드 생성 경로 | 목표 입력 → 프롬프트 조립 → 스트리밍 생성 → 스키마 검증 |
| 턴 처리 루프 | 답변 수신 → 판정 → 프로빙 분기 → 발화 생성 |
| 실시간 분석 | 답변 수신과 동시에 버킷 분류 + 근거 스팬 추출 |
| 세션 종료 후 | 세션 요약 → N 도달 시 질문별 종합 요약 |
| 모델 분리 | 판정은 소형 / 발화는 대형 — 호출 빈도가 다르기 때문 |

**비주얼**: 좌→우 시퀀스 다이어그램 또는 플로우차트.

> 발표 팁: 판정 임계값(확신도 0.6, 격차 0.15) 같은 상세는 여기서 말하지 말 것. S8에서 클립과 함께 나온다. 여기서는 **순서와 분기 구조**까지만.

## S6. 기능 ① 리서치 목표 한 줄 → 인터뷰 가이드 자동 생성

**클립 (40초)**: 목표 입력 → [Generate] → 질문이 하나씩 스트리밍으로 생성 → 버킷 인라인 편집

**설명**
- 목표 1~3줄만 넣으면 질문 5~8개 + 질문별 응답 버킷 4~6개 + 프로빙 규칙이 나온다
- 질문 단위 재생성 — 전체 재생성은 리서처의 편집을 날린다
- 생성 근거(rationale)를 접힌 상태로 노출해 검토를 돕는다

**★ 설계 포인트 — 응답 버킷**
> 각 질문의 응답 분류 카테고리를 **미리** 만든다. 정성조사 코드북을 사전에 만드는 것. 사후 코딩이 2~4주 걸리는 걸 종료 즉시 집계로 바꾼다.
> 그리고 버킷은 분석 라벨일 뿐 아니라 **모더레이터의 실시간 목표**이기도 하다 — 이건 S8에서 드러난다.

**하단 지표**: 생성 P95 ≤ 20초

## S7. 기능 ② 지식 팩 — 검색은 가이드를 만들 때 한 번만, 대화 중엔 하지 않는다

> **(계획 → 구현 반영, 2026-07-23)** 이 슬라이드는 최초 PRD의 "엔티티 카드 검토·승인 게이트" 구상이 아니라, 실제로 구현된 2단 검색(코사인 → 리랭커) + 근거 자동 주입 방식으로 다시 썼다. 엔티티별 위험도 표시·수동 승인 UI는 만들지 않았다(라이트 판 스코프 컷) — 사람이 지키는 게이트는 뒤에 남은 자유편집 금칙어 목록 하나다.

**클립 (25초)**: 브리프 입력 후 [가이드 생성] 클릭 → (백그라운드) 프로젝트 자료 풀 + 전역 페르소나 풀 검색·리랭크 → 근거가 반영된 질문·어휘·응답 버킷이 스트리밍으로 출력

**설명**
- 가이드 생성 시 프로젝트 자료(`briefing_chunks`)와 project 무관 전역 지식 풀(`knowledge_chunks`, 합성 페르소나 등)을 각각 검색한다 — 코사인으로 후보를 넓게 뽑고 리랭커(Qwen3-Reranker-8B)로 top-k를 재정렬하는 2단 검색
- 검색 결과는 "[브리프 검색 근거]"·"[대상 청중]" 블록으로 가이드 생성 프롬프트에만 주입된다. 지어내거나 특정 답을 유도하지 말라는 지시가 함께 붙는다
- 검색·리랭크 실패, 코퍼스 없음은 전부 무동작(fail-open)으로 흡수한다 — RAG 장애가 가이드 생성 자체를 막지 않는다
- 진행자가 먼저 꺼내면 안 되는 주제·표현은 이 검색과 별개로, 리서처가 가이드 화면에서 직접 관리하는 금칙어 목록으로 막는다

**★ 설계 포인트 — RAG 발동 시점을 하나로 고정한다**
> 최초 설계는 인터뷰 도중 모르는 용어가 나오면 그 자리에서 자료를 찾아 이해에만 쓰는 라이브 RAG였다. 그런데 대화 중 검색은 그 순간 무엇이 검색되어 발화에 실렸는지 매 턴 감사해야 하는 표면을 만든다. 그래서 검색 시점을 가이드를 만드는 순간 하나로 앞당겼다 — 그 뒤 인터뷰가 진행되는 동안에는 검색 자체가 없으므로 새어나갈 것도 없다.

**하단 지표**: 진행자 발화의 지식팩 표현 누출률(n-gram 오버랩) 0% — 골드셋/실측은 S10 참조

## S8. 기능 ③ 참가자 인터뷰 — 얕은 답변에 자동으로 파고든다

**클립 (60초)** — 이 발표의 하이라이트
- 링크 진입 → 동의 1클릭 → 대화 시작 (앱 설치·로그인 없음)
- 참가자가 "그냥 편해서요" 같은 얕은 답변 → **프로빙 발동**
- 되물음 이후 구체적 에피소드가 나오는 장면까지

**설명**
- 모바일 웹 320px부터 동작. 링크 배포가 유통 모델의 전부라 앱 설치를 요구하지 않는다
- 8~12분 대화. 프로빙은 진행률에 반영하지 않는다 — "끝이 안 난다"고 느끼면 이탈한다
- 참가자 화면에 리서치 목표·클라이언트 브랜드를 노출하지 않는다. 조사 의도를 추론하면 답변이 편향된다

**★ 설계 포인트 — 프로빙 발동 조건**
> S5의 턴 루프가 실제로 무엇을 보고 판단하는지가 여기다.
> 최고 버킷 확신도가 **0.6 미만**이거나 상위 2개 버킷 격차가 **0.15 미만**이면 프로빙이 발동한다. S6에서 만든 버킷이 여기서 대화의 종료 조건으로 작동하는 것이다.
> 프로빙 5종: 심화 · 예시요청 · 대비 · 결과추적 · 감정원인
> 단, 가이드 밖으로는 나가지 않는다. 흥미로워 보여도 AI가 새 주제를 개시하지 않는다.

**하단 지표**: 프로빙 후 정보량 3배 · off-guide 발화 0건

## S9. 기능 ④ 분석 대시보드 — 종료 즉시 구조화되어 있다

**클립 (45초)**: 대시보드 진입 → 질문별 컬럼 그리드 → 버킷 분포 → 요약 문장 hover → 근거 발화 하이라이트 → 원문 트랜스크립트 점프

**설명**
- 질문 = 컬럼, 참가자 = 행. 한 질문에 대한 전원의 답변을 세로로 훑는다
- 분석은 배치 잡이 아니라 세션 종료와 동시에 완료된다
- 버킷 배정은 `{bucket_id, confidence, evidence_span}`으로 저장되고 리서처가 수동 교정할 수 있다

**★ 설계 포인트 — 근거 없는 문장은 생성하지 않는다**
> 요약의 모든 주장은 특정 발화로 역추적 가능해야 한다. 근거 없는 문장은 생성 자체를 차단한다.
> 그리고 "대부분", "많은" 대신 **"27명 중 18명(67%)"** 처럼 수치로 쓴다. 정성조사를 정량으로 말할 수 있게 하는 것이 이 제품의 목적이기 때문이다.

**하단 지표**: 종료 → 리포트 P95 ≤ 5분

# 검증 · 성과 블록 (S10–S12) — 설계 노트

## 이 블록의 문제

목표 수치만 나열한 표는 **결과가 아니라 계획**이다. 심사자가 "그래서 지금 얼마나 나왔나요"라고 물으면 무너진다. 이 블록은 반드시 **실측 여부에 따라 다르게 만들어야** 한다.

### 분기 판단

| 상황 | S10·S11 처리 |
|---|---|
| **A. 실측 데이터 있음** (일부라도) | 측정된 수치를 결과로 크게, 미측정은 회색으로 명시 |
| **B. 하네스만 구축** (골드셋·CI·벤치마크 스크립트) | 슬라이드의 주어를 "목표"에서 **"우리가 만든 측정 장치"**로 바꾼다. 장치 자체가 산출물 |
| **C. 둘 다 없음** | **슬라이드를 지운다.** 측정하겠다는 약속만 있는 슬라이드는 없는 것보다 나쁘다 |

> 대부분의 프로젝트는 B다. B는 부끄러운 게 아니라 오히려 강한 카드다 — "품질을 어떻게 측정할지 정의하고 파이프라인까지 짰다"는 건 대다수 팀이 안 하는 일이다. 단, **B를 A인 것처럼 보이게 만들면 그 순간 신뢰를 잃는다.**

## S10. 품질 검증 — 무엇을, 어떻게 재는가

### 구성 (분기 B 기준)

**좌: 측정 대상 3분류**

| 축 | 대표 지표 | 방법 |
|---|---|---|
| 생성 품질 | 유도 질문 비율, 버킷 MECE 위반율 | 전문가 라벨링 셋 + LLM 심판 |
| 분류 품질 | 버킷 분류 정확도 (Cohen's κ) | 인간 코더 **이중 라벨링 골드셋 500건** |
| 안전 게이트 | Knowledge leak rate, 요약 환각률 | 발화×팩 n-gram 오버랩, 근거 스팬 검증 |

**우: 현재 상태 (솔직하게)**

| 항목 | 상태 |
|---|---|
| 골드셋 구축 | 완료 / N건 |
| CI 연동 | 완료 / 미완 |
| 실측 라운드 | 진행 중 / 예정 |

**★ 강조할 것 — 게이트 개념**
> Knowledge leak rate는 목표가 아니라 **릴리즈 게이트**다. 0%가 아니면 팩 기능을 출시하지 않는다. 지표를 만들어놓고 안 지키면 의미가 없으므로, 미달 시 출시를 막는 규칙까지 같이 정의했다.

**비주얼**: 좌 3분류 카드 / 우 상태 체크리스트. 게이트 지표만 다른 색.

> 발표 팁: 숫자가 있으면 숫자를, 없으면 **"골드셋 500건을 어떻게 만들었는가"**를 말하라. 방법론 설명이 빈 목표표보다 훨씬 설득력 있다.

## S11. 추론 비용 — 저전력 NPU 검토

### 왜 이 슬라이드가 필요한가

과금은 **리서치 질문 단위**인데 프로빙·분류·요약은 전부 비과금이다. 즉 토큰 대부분이 매출이 붙지 않는 원가다. 여기서 원가 구조를 잡지 못하면 프로빙을 줄이게 되고, 그러면 제품의 핵심 가치가 사라진다.

### 구성

**1. 워크로드 특성 (이게 핵심 논점)**
> 참가자가 타이핑하는 동안 가속기는 논다. duty cycle이 2% 미만이라 **총 에너지를 유휴 전력이 지배한다.** 그래서 일반적인 "초당 토큰" 비교로는 실제 비용이 안 드러난다.

**2. 측정 지표 3개**

| 지표 | 조건 | 역할 |
|---|---|---|
| 카드당 SLA 충족 동시 세션 수 | P95 ≤ 2초 유지 | 용량 |
| 세션당 Wh (벽면, idle 포함) | 24h 도착 분포 리플레이 | 비용 근거 |
| 버킷 분류 κ 열화 | GPU 대조군 대비 Δ | 통과 조건 |

**3. 결론 형태**
> "몇 배 빠르다"가 아니라 **"월 N세션을 넘으면 유리하다"** — 손익분기 세션 수

**4. 현재 상태 (2026-07-23)**
> 의뢰자 화면에 벤치마크 대시보드(`/projects/benchmark`)를 만들었다 — M1(카드당 SLA 동시 세션)·M2(세션당 Wh, idle 분해)·M3(버킷 분류 κ 게이트)·손익분기 S\* 카드와 24h 전력 시계열까지 레이아웃은 완성. **측정 하네스(부하 재생기·PDU/Prometheus 수집)는 아직 실행 전이라 화면의 모든 수치는 `—`(null)로 뜬다** — 예시 숫자로 채우지 않았다.

**비주얼**: 손익분기 곡선 1개. x축 월 세션 수, y축 월 비용, 교차점에 N 라벨. (측정 전이면 대시보드의 null 상태 화면을 그대로 캡처 — 예시 곡선을 그리지 않는다)

> 발표 팁: 측정 미완료면 **"측정 설계 완료 / 대시보드 구현 완료 / 실행 예정"**을 슬라이드에 박아둘 것. tok/s·tok/s/W·TDP 비교는 이 벤치마크의 명시적 비목표다(계측 스펙 §0) — 절대 등장시키지 않는다. 추정치를 실측처럼 그리면 질의응답에서 가장 먼저 무너지는 지점이다.

## S12. 실행 결과 — 직접 돌려본 것

> 출시를 전제로 한 목표 수치는 이 발표에서 의미가 없다. 대신 **지금 실제로 측정된 값**만 올린다.
> 셀프 테스트 인터뷰(F3.4)와 합성 참가자 시뮬레이션(F3.4.2)을 돌리면 실제 참가자 없이도 나오는 수치가 있다.

### 잴 수 있는 것 (실측 — 이 슬라이드의 본문)

| 항목 | 무엇을 보여주나 |
|---|---|
| 실행한 세션 수 | 자체 테스트 N건 / 합성 참가자 M건 |
| 세션 평균 소요 시간 | 설계값 8~12분과 실제의 차이 |
| 세션당 프로빙 발생 횟수 | 프로빙 엔진이 실제로 작동하는가 |
| 기타(catch-all) 버킷 비율 | 버킷 설계가 응답 공간을 덮는가 |
| 버킷 커버리지 | 아무도 안 걸린 버킷이 있는가 |
| 가이드 생성 지연 / 모더레이터 응답 지연 | P95 실측 |

### 못 재는 것 (명시할 것)

참가자 완료율 · 리서처 재사용률 · NPS — **실제 사용자가 있어야 나오는 값이므로 측정하지 않았다고 밝힌다.**

### 강조 포인트

숫자 자체보다 **"돌려봤더니 설계와 달랐던 것"** 하나를 찾아서 말하는 게 훨씬 강하다. 예를 들어 프로빙이 예상보다 자주 터졌다거나, 특정 버킷에 아무도 안 걸렸다거나. 그 발견이 곧 S13의 미해결 질문으로 이어진다.

**비주얼**: 실측 수치 4~6개를 숫자 카드로. 못 잰 항목은 하단에 회색 한 줄로 작게.

> 발표 팁: S10에서 "하네스는 만들었다"고 말했다면, 이 슬라이드가 그 하네스로 실제로 돌린 결과다. 두 장이 짝을 이루면 "설계만 했다"는 인상이 사라진다.
>
> 참고: 가치 명제인 "리서치 리드타임 3주 → 3일"은 실측이 아니라 설계 목표이므로 여기가 아니라 **S3 또는 S13**에서 말한다. 결과 슬라이드에 목표를 섞지 말 것.

## 클립 목록

| 클립 | 슬라이드 | 길이 | 내용 |
|---|---|---|---|
| clip1 | S6 | 40초 | 가이드 생성 스트리밍 |

| clip4 | S9 | 45초 | 분석 대시보드 드릴다운 |
| **merged** | 제출용 | 2분 50초 | 위 4개를 순서대로 이어붙인 통합본 |

> **merged 파일을 반드시 별도로 만들 것.** 제출 요건이 "동영상 파일 또는 동영상이 임베딩된 ppt"이므로, 클립을 쪼갠 상태로만 두면 단독 영상 제출물이 없어진다.

## 제작 규칙

- **자동재생 + 루프 + 무음.** 발표 중 클릭 조작이 들어가면 흐름이 끊긴다
- **자막 필수.** 발표장에서 영상 소리를 틀지 않는다는 전제로 만든다
- clip3의 프로빙 발동 순간은 화면 강조(하이라이트 박스 또는 순간 정지)를 넣는다. 이 장면 하나가 제품의 핵심이라 놓치면 발표가 실패한다
- 실패 케이스는 넣지 않는다. 2분 50초에 담을 수 없다
- 각 클립은 앞뒤 0.5초 여백을 두고 자른다

## 백업

- 클립 임베딩 실패에 대비해 각 기능 슬라이드에 **정지 스크린샷 1장**을 숨김 레이어로 준비
- merged 파일을 USB와 로컬에 각각 보관

- ★ 설계 포인트는 배경 틴트 카드로, 4장에서 위치까지 같게
- 하단 지표는 같은 높이, 같은 크기

**금지**
