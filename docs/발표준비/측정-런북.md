> 사고시간이 들어가면 그림이 달라진다. 그 차이를 보이는 게 이 벤치마크의 존재 이유다.

## 1. 왜 재측정인가 — 수치가 낡았다

2026-07-23 측정 이후 파이프라인이 두 번 바뀌었고, **둘 다 M4 와 M1 을 직접 건드린다.**

| 변경 | 무엇이 바뀌나 | 영향받는 수치 |
|---|---|---|
| **감정 태깅 병렬화** (PR #30) | 직렬 3단계 → 감정 태깅이 질문 생성과 겹침 | 턴 전체 지연, 단계 비중 |
| **가드레일 판정 규칙 수정** (PR #33) | 콜백을 유도로 오판하던 것 제거 | **재작성률**, 가드레일 단계 지연, 턴 전체 |

따라서 아래 수치는 **현재 성능이 아니다.** 발표·보고서에 쓸 때는 반드시 "수정 전" 이라고 적는다:

- 턴 전체 p50 `5.52s` · 부하 0 기준 `4.5s`
- 가드레일 `3.02s` (턴의 55%)
- **재작성률 `85.4%`** (522턴 중 446턴)
- 감정 태깅 `0.58s` (직렬 기준)

> 재측정에서 **재작성률이 얼마나 떨어졌는지**가 이번 수정의 성패다. 그 숫자 하나가
> "원인을 특정하고 고쳤다"의 증거가 된다. 안 떨어졌으면 진단이 틀린 것이고, 그것도 결과다.

## 2. 재야 하는 것 — 실행 순서

### 2-1. 준비 (런 전에 한 번)

- [ ] **벤치 스크립트를 리포로 옮긴다.** 현재 `bench_*.py`·`interview_sim.py` 가 팟에만 있어 재현이 불가능하다. 이게 안 되면 아래 결과도 "재현 불가"로 남는다.
- [ ] **실행 메타데이터 덤프**를 런 시작 시 자동 기록: `sdk_version` · `firmware_version` · `driver_version` · `model_id` · `quantization` · `governor` · `prefix_caching` · `tensor_parallel_size` · `corpus_hash` · `prompt_template_hash` (스펙 §4)
- [ ] **턴 레코드에 벽시계 시각을 턴마다** 남기게 고친다 — `t_request` · `t_first_token` · `t_last_token`. 지난 런은 레벨 종료 시 일괄 기록이라 순간 동시 생성을 검증할 수 없었다.
- [ ] **모델 배치를 기록한다.** 지난 런은 전 역할 `Qwen3-8B` 였다(스펙 §5의 EXAONE-32B 배치가 아님). 이번에도 8B 면 그렇게 적는다 — 배치가 다르면 수치를 같은 표에 섞지 않는다.
- [ ] idle 베이스라인 측정 (부하 0, 서버 기동 상태, 10분)

### 2-2. 본 측정 (M1 · M1-b · M2 · M4 를 한 런에서)

세 지표는 **같은 런에서 나온다.** 따로 돌리지 않는다.

- [ ] 동시 세션 슬롯 `C = 8 / 16 / 32` 각각
  - [ ] 워밍업 3분 폐기 → 본 측정 10분
  - [ ] 기록: 턴 수 · 실패 수 · `p50` · `p95` · `ttft_p95`
  - [ ] 기록: 처리량(턴/분) → **M1-b** 평균 동시 생성 = 처리량 × 평균 턴 지연
- [ ] 같은 런 동안 **카드 전력 시계열** 수집 (`furiosa_npu_hw_power`, 1s 간격)
- [ ] 같은 런 동안 **단계별 지연** 수집 → **M4**
  - [ ] 감정 태깅 / 질문 생성 / 가드레일(판정·재작성) 각각 p50
  - [ ] **재작성 발생률** — 이번 런의 핵심 관전 포인트
  - [ ] 감정 태깅은 이제 병렬이므로 **벽시계 기여도**로 보고(직렬 합과 다르다고 명시)
- [ ] `furiosa_npu_throttling_events_count` 증가 구간이 있으면 그 회차 무효 → 재측정

### 2-3. 정성 관찰 (자동화 불가, 눈으로 봐야 함)

- [ ] 인터뷰 시뮬 10턴 로그를 **읽는다**
  - [ ] 재작성된 질문이 구체성을 유지하는가 (지난 런: "그 부분 어떻게 보세요?" 로 뭉개짐)
  - [ ] 대화가 한 주제에 갇히지 않고 가이드 후반 문항까지 가는가
  - [ ] `정상종료` 판단이 되는가 (지난 런: 10턴 상한까지 못 끝냄)

> 이건 지표가 아니라 **관찰**이다. 숫자로 만들지 말고 인용 몇 줄로 남긴다.

### 2-4. 여유 있으면

- [ ] 프리픽스 캐시 히트율 — furiosa-llm 서버 로그 파서 구현 후 수집
- [ ] governor `PowerSave` 고정 부하(M1의 70%) 전력 비교
- [ ] prefix caching `off` 대조

## 3. 재지 않는 것 — 그리고 그 이유

**"아직 안 했다"가 아니라 "이 환경에서는 못 한다"이다.** 물어보면 이렇게 답한다.

| 항목 | 답변 |
|---|---|
| **벽면 PDU 전력** | 공유 팟이라 PDU에 물리적으로 접근할 수 없다. 카드 센서로 재되 **CPU·NIC·팬 미포함 하한값**이라고 명시한다. |
| **GPU 대조군** | 대조군 하드웨어가 없다. 그래서 "몇 배 빠르다"를 말하지 않는다. |
| **Cohen's κ** | 골드셋 500건(인간 이중 라벨링)이 없고, 대조군이 없어 Δκ 자체가 정의되지 않는다. 분류 품질은 **미검증**이라고 적는다. |
| **손익분기 S\*** | 대조군 비용과 벽면 전력에 둘 다 의존한다. 추정 상수로 곡선을 그리면 그 순간 실측 문서가 아니게 된다. 대신 **idle 지배 분석**으로 같은 논증을 한다. |
| **tok/s · tok/s/W · TDP** | 스펙 §0이 비목표로 못박았다. 이 워크로드의 본질(저부하·긴 컨텍스트·짧은 출력)과 무관한 지표다. |
| **"vs L40S 267.8%"** | 우리 측정이 아니라 교재 인용이다. 슬라이드에 올리면 우리 성과로 오인된다. |

## 4. 기록 필수 — 이게 빠지면 결과가 못 쓴다

런마다 아래를 **결과 파일과 같은 폴더에** 남긴다.

1. **실행 메타데이터** 전량 (§2-1)
2. **모델 배치** — 역할별로 어떤 모델을 썼는지
3. **측정하지 않은 항목과 사유** — §3 표를 그대로 복사해 넣는다
4. **원자료 경로** — CSV·로그 파일명
5. **코드 리비전** — `git rev-parse HEAD`. 어떤 커밋의 파이프라인을 쟀는지가 재작성률 해석을 좌우한다
